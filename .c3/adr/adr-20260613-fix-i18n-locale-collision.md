## Goal

Remove `"en"` from the Astro i18n `locales` array in `astro.config.ts` so that the language-filter URL segment `/posts/lang/en` is never interpreted as an i18n UI locale, preventing Astro from setting `Astro.currentLocale` to `"en"` on every page served under that path.

## Context

The blog at `/posts/lang/en` (route `src/pages/posts/lang/[lang]/[...page].astro`) uses the URL segment `en` as a **content-language filter parameter**, not as a UI locale prefix. However, Astro's i18n subsystem scans every URL path segment for values matching the configured `locales` array (`["vi","en"]`). When a visitor is on `/posts/lang/en`, the segment `en` matches the configured locale `"en"`, so Astro sets `Astro.currentLocale === "en"` for the entire page render.

Every shared component (`LangFilter.astro`, `Header.astro`, `Breadcrumb.astro`, `Main.astro`, `Tag.astro`, `Card.astro`) and every page template that renders on this route calls `getRelativeLocaleUrl(Astro.currentLocale ?? config.site.lang, ...)`. With `currentLocale === "en"` (a non-default locale since `prefixDefaultLocale: false` is set), `getRelativeLocaleUrl` emits `/en/...`-prefixed URLs. No `/en/` routes are ever generated by the static build — the blog has no fully localized UI shell under `/en/`. The result: every navigation and filter link on `/posts/lang/en` is a 404. `/posts/lang/vi` is unaffected because `vi` is the default locale and `prefixDefaultLocale: false` means the default locale emits no prefix.

**Confirmed evidence (before fix):** `grep -oE 'href="[^"]*"' dist/posts/lang/en/index.html` shows `href="/todd.lamb/en/"`, `href="/todd.lamb/en/posts/"`, `href="/todd.lamb/en/posts/lang/vi/"`, `href="/todd.lamb/en/posts/lang/en/"`, `href="/todd.lamb/en/about/"`, `href="/todd.lamb/en/archives/"`, `href="/todd.lamb/en/search/"`. None of these paths exist in the built `dist/`. Equivalent links on `/posts/lang/vi/index.html` use `/todd.lamb/posts/...` (correct).

The root issue is a **namespace collision**: the content-language filter param `[lang]` in `posts/lang/[lang]/[...page].astro` reuses locale codes (`vi`, `en`) as URL values, and Astro's i18n routing intercepts those same values as UI locale signals.

## Decision

Remove `"en"` from `i18n.locales` in `astro.config.ts`, changing `locales: ["vi", "en"]` to `locales: ["vi"]`.

**Rationale:** This blog is Vietnamese-first with multilingual *post content* (posts tagged with `lang: vi` or `lang: en`). It is NOT a fully localized multi-locale UI — there are no `/en/` pages, no `/en/` layout routes, and no UI string switching based on URL locale. The `i18n.locales` config was set to `["vi","en"]` to support `useTranslations(Astro.currentLocale)`, but `useTranslations` already falls back to English internally (`translations[locale] ?? translations["en"]`) regardless of what `Astro.currentLocale` returns. Removing `"en"` from `locales` means Astro will never treat the `en` URL segment as a UI locale, so `Astro.currentLocale` returns `undefined` on `/posts/lang/en` and every `getRelativeLocaleUrl(locale, ...)` call falls through to `config.site.lang === "vi"`, emitting correct unprefixed `/posts/...` links.

**Safety checks completed:**

1. `astro-paper.config.ts` `site.lang` is `"vi"` — the `?? config.site.lang` fallback resolves to the correct default.
2. `useTranslations` defaults to `"en"` when locale is missing and falls back to `translations["en"]` — unaffected by removing `"en"` from `i18n.locales`.
3. `Datetime.astro` uses `locale === "vi"` check for date formatting — with `currentLocale` returning `undefined` on the filter page, `locale` becomes `config.site.lang = "vi"`, so Vietnamese date format applies site-wide (correct and intended).
4. No code path gates on `Astro.currentLocale === "en"` explicitly.
5. The `/posts/lang/en` route still generates correctly — it is a dynamic `[lang]` param route, not an Astro i18n route.

## Alternatives Considered

| Alternative | Rejected because |
| --- | --- |
| Change filter route to non-locale URL segments (e.g. /posts/lang/vietnamese and /posts/lang/english) | Requires changes to 3+ files (route file, LangFilter.astro, getPostLanguages util, any URL constructors). More invasive and makes filter param values in URLs less clean than the two-letter lang codes. The root issue is Astro treating the param as a locale, which is more cleanly fixed at the config level. |
| Override Astro.currentLocale per-page via middleware | Astro does not expose a public API to override currentLocale; it is derived from the URL by the framework. Middleware approaches would require fragile header injection before i18n middleware runs. |
| Remove all uses of Astro.currentLocale in favor of explicit locale prop drilling | Would require touching 15+ files (every component and page template). Disproportionate churn for a single-locale blog where the fix is a one-character config change. |
