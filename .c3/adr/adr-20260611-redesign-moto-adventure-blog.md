## Goal

Rebrand and redesign the existing AstroPaper v6 personal blog (todd.lamb repo, "Hiep Lam" site) into a Vietnamese adventure-motorcycle travel blog: replace the generic AstroPaper demo identity (title, hero copy, color tokens, card presentation) with a moto-adventure visual identity; replace the 11 AstroPaper demo posts with 45 original Vietnamese blog posts about adventure motorbike trips (the owner trimmed the planned 50 to 45 mid-implementation; scripts/trips.json is the source of truth), each containing locally generated SVG scenic images; ship an optional HTTP Basic Auth gate (owner decision post-merge: the site is PUBLIC by default; setting AUTH_PASSWORD turns the gate on without code changes); and deploy the result as a Docker image via Dokploy. The decision being authorized: full content + visual redesign within the existing Astro 6 static architecture, plus an nginx-level authentication layer in the runtime Docker stage, with Dokploy as the deployment target.

## Context

The repo is AstroPaper v6 (Astro 6 + Tailwind 4 + Bun), personalized only in name ("Hiep Lam" in astro-paper.config.ts) but still shipping AstroPaper demo content (11 theme-documentation posts under src/content/posts/), the stock "Mingalaba" hero in src/pages/index.astro, stock theme colors in src/styles/theme.css, and AstroPaper socials/editPost URLs. The owner wants a real blog: 50 image-rich posts about adventure motorcycle journeys, a matching redesign, and a private deployment on a Dokploy server where users log in to read. Constraints: the site is fully static (no server runtime — auth must live in the serving layer, not in Astro); the Dockerfile already builds with Bun and serves dist/ via nginx:mainline-alpine-slim, so auth belongs in that nginx stage; content must satisfy the Zod schema in src/content.config.ts (ref-content-schema); colors must flow through CSS custom properties in src/styles/theme.css (ref-tailwind-design-system); UI strings go through useTranslations (ref-i18n-strategy); dynamic per-post OG images are generated by the satori+sharp pipeline (ref-og-image-generation) and must keep working for 50 new posts. Topology involved: container c3-1 (Hiep Lam Blog) and components c3-101 content-collections, c3-102 layouts, c3-103 ui-components, c3-104 styles, c3-110 pages-routing, c3-112 og-image, c3-106 ci-release. The local machine has no Docker and no Dokploy credentials; shell history shows the owner installs Dokploy via its official script, so the deploy step targets the owner's Dokploy server and final deploy verification requires server URL + credentials from the owner ([ASSUMED] blocking input, requested at deploy time).

## Decision

Implement everything inside the existing AstroPaper architecture — no framework or adapter change. Concretely: (1) Redesign via design tokens + targeted component edits: new adventure palette (light: warm sand background with burnt-orange accent; dark: deep night-blue with amber accent) expressed only as CSS custom properties in src/styles/theme.css per ref-tailwind-design-system; rewrite the hero section of src/pages/index.astro with moto-adventure copy and brand; extend src/components/Card.astro to render a per-post hero image thumbnail; extend src/layouts/PostLayout.astro to render the post hero image above the article; update astro-paper.config.ts (title, description, socials, editPost disabled, posts.perPage/perIndex raised) and the about page content; switch the UI chrome to Vietnamese by adding src/i18n/lang/vi.ts and making vi the astro.config defaultLocale per ref-i18n-strategy, and move display/body typography to Archivo (vietnamese subset) while keeping Google Sans Code for code blocks and adapting the OG pipeline (getFontSourcesByWeight) to feed all font subsets to satori. (2) Content: delete the 11 demo posts and demo assets; add 45 Vietnamese .md posts (owner-trimmed from 50) (English slugs) under src/content/posts/, each with required frontmatter (title, description, pubDatetime, tags) plus 2–3 inline images; images are deterministic, locally generated SVG scenic illustrations (layered mountain/road/sky compositions, seeded per post) produced by a one-shot Bun script scripts/generate-post-images.ts writing to public/images/posts/<slug>/, so no external image dependencies, no copyright risk, and reproducible builds. (3) Auth: keep the static build; add an optional HTTP Basic Auth gate in the nginx runtime stage — docker/default.conf includes /etc/nginx/auth-gate.conf site-wide and the /docker-entrypoint.d/30-htpasswd.sh hook decides per start: AUTH_PASSWORD unset writes an empty gate (public site, the default per owner decision after review), AUTH_PASSWORD set generates /etc/nginx/.htpasswd (openssl passwd -apr1, user AUTH_USER default "rider") and enables auth_basic. This wins over an Astro Node-adapter session login because the site stays static (faster, simpler, no DB) and Dokploy env vars manage credentials. (4) Deploy: Dokploy application built from this repo's Dockerfile (git provider), AUTH_USER/AUTH_PASSWORD set as Dokploy environment variables; deploy executed via Dokploy API/UI once the owner supplies server URL + token. Work lands as one feature branch in a git worktree, PR'd and squash-merged to main per repo workflow.

## Alternatives Considered

| Alternative | Rejected because |
| --- | --- |
| Astro Node adapter + session-based login page | Converts a fully static repo (Dockerfile serves dist/ via nginx) into a server runtime, adds session storage and attack surface for a single-tenant private blog; nginx basic auth meets the "log in to view" requirement with zero app-code changes |
| Stock photos / external image CDN (Unsplash etc.) for post images | Build would depend on network + third-party licensing; 50 posts × multiple images is rate-limit-prone; deterministic local SVGs are license-free, tiny, and reproducible in CI and the Docker build |
| New theme/framework swap (e.g. rebuild on another Astro theme) | Existing AstroPaper v6 architecture is C3-documented with working OG, search, i18n, CI; redesign via tokens + targeted components reuses all of it at a fraction of the risk |
| Deploy via Cloudflare Pages (site.url currently points at astro-paper.pages.dev) | Owner explicitly requires Dokploy as the target; Cloudflare Pages also cannot enforce the basic-auth gate without Workers code |

## Risks

| Risk | Mitigation | Verification |
| --- | --- | --- |
| 50-post build inflates build time or breaks pagefind/OG generation | Posts are plain .md with small SVG assets; build run locally before PR; OG endpoint smoke-checked | bun run build completes; dist/posts/<slug>/index.png exists |
| Basic auth blocks healthchecks or asset loading on Dokploy | nginx exposes /healthz location with auth off for healthcheck; everything else gated | curl /healthz → 200 unauthenticated; / → 401 |
| Vietnamese content with en locale breaks i18n typing | Post bodies are content (not UIStrings); site lang stays "en"; no locale files touched | astro check passes |
| Owner expects a gated blog but forgets AUTH_PASSWORD, leaving the site public | Container start logs an explicit '[auth] serving the blog publicly' line; README documents the toggle | docker logs show the [auth] line; README Docker section |
| Dokploy server details unavailable locally | All deployable artifacts (Dockerfile, env contract) land in repo; deploy step asks owner for server URL/token via question tool when reached | Deployment reachable and login-verified in browser |
